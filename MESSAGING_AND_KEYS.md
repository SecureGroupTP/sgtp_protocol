# Сообщения, Ключи и Ротации

## CHAT_REQUEST и выдача CK

1. Немастер после handshake отправляет `CHAT_REQUEST` мастеру.
2. Мастер валидирует запрос.
3. Мастер генерирует новый `Chat Key` и рассылает `CHAT_KEY` юникастом каждому участнику.
4. Участники отвечают `CHAT_KEY_ACK`.

## Ротация ключа

- Периодическая ротация: каждые `180s`.
- Во время окна ротации (`до получения всех ACK`) мастер отклоняет входящие `MESSAGE`.
- На отклонение мастер шлёт `MESSAGE_FAILED` участникам.

## MESSAGE

- Broadcast пакет.
- Шифрование текущим `CK`.
- `message_uuid` и `nonce` открытые.
- Подпись обязательна.

## STATUS и MESSAGE_FAILED

- `STATUS` и `MESSAGE_FAILED` шифруются pairwise ключом между sender/receiver.
- В `v1` nonce берётся из `timestamp`.
- В Flutter `v2` поддерживается явный nonce в начале payload для shared-key пакетов.
- В текущем runtime Flutter-клиента принимаются кадры только с `version=0x0002`, поэтому практический путь совместимости — использовать `v2` формат.

## FIN

- `FIN` broadcast, аутентификация через `CK`.
- Payload содержит nonce и AEAD-тег для пустого plaintext.
- После FIN мастер запускает новую ротацию CK для оставшихся.
