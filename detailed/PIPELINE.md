# Детальный пайплайн передачи сообщений

Этот документ описывает полный путь каждого типа сообщения — от момента, когда пользователь нажимает «Отправить», до отображения у получателя. Охватываются текст, изображения, голосовые, видео, видеозаметки и произвольные файлы.

---

## 1. Архитектурная модель

SGTP — peer-to-peer протокол с опциональным ретранслятором. Важно понимать несколько ключевых свойств, которые влияют на весь пайплайн:

**Relay — почти тупой форвардер.** Сервер не знает о содержимом сообщений, не проверяет подписи и не буферизует доставку. Исключение: для входящего `MESSAGE` relay читает первые 16 байт payload (`message_uuid`) и отправляет обратно отправителю `MESSAGE_ACK`. Если получатель оффлайн — пакет теряется. Исторические сообщения хранятся только у peer-ов, а не на сервере.

**Каждый пакет подписан и (или) зашифрован.** Подпись ed25519 защищает целостность и авторство каждого фрейма. Содержательная часть (payload MESSAGE-пакетов) шифруется симметричным групповым ключом (Chat Key, CK).

**Мастер управляет ключами.** Роль мастера получает peer с минимальным UUID в комнате. Только он генерирует и распространяет Chat Key, решает конфликты в момент ротации и исключает участников по запросу.

**Медиа-данные не живут на сервере.** Нет CDN, нет object storage. Все байты проходят через relay как обычные зашифрованные MESSAGE-фреймы — нарезанными на чанки по 100 КиБ (по умолчанию).

## 2. Жизненный цикл соединения (до первого сообщения)

### 2.1 Транспортный уровень

Клиент выбирает один из шести вариантов транспорта:

| Транспорт | Когда используется |
|---|---|
| TCP (raw) | Нативные приложения, нет ограничений файрвола |
| TCP + TLS | Нативные приложения, нужна защита на уровне транспорта |
| HTTP / HTTPS | Среды без поддержки постоянных соединений; polling |
| WebSocket / WSS | Браузеры, окружения с HTTP-прокси |

При TCP-подключении сервер немедленно отправляет **25-байтный discovery-баннер** — бинарное описание доступных портов. Flutter-клиент читает и отбрасывает эти 25 байт перед началом SGTP frame stream.

При использовании HTTP-транспорта клиент сначала получает discovery через `GET /sgtp/discovery` (JSON с перечнем транспортов и портов), затем открывает сессию через `POST /sgtp/session`.

Для Flutter discovery делается с fallback-порядком:
1. `https://<host>:443/sgtp/discovery`
2. `http://<host>:80/sgtp/discovery`
3. `http://<host>:77/sgtp/discovery`

### 2.2 Handshake-фаза

Полный хендшейк с нуля до состояния READY занимает 500–800 мс при нормальном latency. Шаги:

```
[Новый peer]                         [Существующий peer / Мастер]

INTENT ──────────────────────────────>
                    <──────────────── PING (x25519_eph_pub, ed25519_pub)
PONG (x25519_eph_pub, ed25519_pub) ──>
  ← вычисляем DH shared secret →       ← вычисляем DH shared secret →

                    <──────────────── INFO_RESPONSE (список UUID участников)

[Допинговываем оставшихся peer-ов]

CHAT_REQUEST ────────────────────────>
                    <──────────────── CHAT_KEY (зашифрован pairwise ключом)
CHAT_KEY_ACK ───────────────────────>

[Состояние: READY — можно отправлять MESSAGE]
```

После получения CHAT_KEY клиент расшифровывает групповой ключ своим pairwise shared secret и сбрасывает счётчик `_myNonce = 0`. С этого момента он может шифровать и отправлять MESSAGE-пакеты.

## 3. Текстовые сообщения

### 3.1 Сторона отправителя

```
Пользователь вводит текст → sendMessage("Привет!")
│
├─ Генерация messageUUID (UUIDv7, 16 байт)
├─ nonce = _myNonce++ (монотонно возрастающий uint64)
│
├─ Сериализация JSON payload:
│   {
│     "v": 1,
│     "type": "text",
│     "text": "Привет!",
│     "pub": "<hex ed25519 pubkey>",
│     "avatar": "<base64 PNG или null>"
│   }
│
├─ UTF-8 encode → plaintext bytes
├─ ChaCha20-Poly1305 encrypt(plaintext, key=ChatKey, nonce=nonce)
│   → ciphertext (plaintext.len + 16 байт тега)
│
├─ Build MESSAGE frame:
│   Header:
│     room_uuid(16) | receiver_uuid=BROADCAST(16 нулей) |
│     sender_uuid(16) | version=0x0002 | type=0x0007 |
│     payload_len(4) | timestamp_unix_ms(8)
│   Payload:
│     message_uuid(16) | nonce(8) | ciphertext(N)
│   Signature:
│     ed25519_sign(header + payload) → 64 байта
│
├─ Постановка в _sendChain (серийная очередь)
├─ Сохранение в локальный SQLite (messageUUID, plaintext, timestamp)
└─ Немедленный emit SgtpMessageReceived в UI (оптимистичное отображение)
```

### 3.2 Relay

Relay читает заголовок (первые 64 байта фрейма), извлекает `room_uuid` и `receiver_uuid`. Поскольку `receiver_uuid` = 16 нулей (broadcast), пакет рассылается всем соединениям в комнате кроме отправителя. Для `MESSAGE` relay дополнительно читает `message_uuid` (первые 16 байт payload) и возвращает отправителю `MESSAGE_ACK`.

### 3.3 Сторона получателя

```
Получен байтовый поток из сокета / WS / HTTP recv
│
├─ tryExtractFrame(buffer): считываем 64-байтный заголовок,
│   читаем payload_len, извлекаем полный фрейм
│
├─ _dispatch(frame):
│   ├─ Проверка timestamp (±30 секунд от now)
│   ├─ Проверка protocol version (`frame.version == 0x0002`)
│   └─ Роутинг по packet_type → _onMessage(frame)
│
├─ _onMessage:
│   ├─ Проверка ed25519 подписи (sender.ed25519PubKey)
│   ├─ Извлечение message_uuid(16) и nonce(8) из начала payload
│   └─ ChaCha20-Poly1305 decrypt(ciphertext, key=ChatKey, nonce=nonce)
│       → plaintext bytes
│
├─ JSON decode → {v, type, text, pub, avatar}
├─ Проверка поля type == "text"
├─ Создание ChatMessage(content: "Привет!", senderUUID: ...)
├─ Сохранение в SQLite
└─ emit SgtpMessageReceived → UI перерисовывает список
```

### 3.4 Особые текстовые типы

Кроме `"type": "text"` через MESSAGE-пакеты передаются и другие текстовые данные:

- **`"chat_meta"`** — обновление имени/аватара peer-а. Broadcast.
- **`"message_read"`** — read receipt (отправляется только в реальном времени, не сохраняется в историю).
- **`"reaction"`** — эмодзи-реакция на конкретное сообщение (`msg_id`, `emoji`, `add: bool`).

## 4. Изображения (Image / GIF)

### 4.1 Особенности медиа-передачи

Медиа-сообщения отличаются от текста одним ключевым свойством: данные нарезаются на чанки, каждый из которых отправляется отдельным MESSAGE-пакетом. Размер чанка по умолчанию — **100 КиБ** (102 400 байт). На веб-платформе минимальный размер чанка увеличивается до **512 КиБ** из-за более высоких накладных расходов на WASM-крипто.

Максимальное число чанков — **9 999**. Теоретический предел одного файла: `9999 × chunk_size`, то есть около 1 ГБ при 100 КиБ чанках. На практике ограничения приходят раньше из-за payload limit 16 МиБ на один фрейм и доступной RAM.

### 4.2 Пайплайн отправки изображения

```
File / XFile путь к изображению
│
├─ Генерация file_id (UUID, уникальный идентификатор медиа)
├─ Открытие файла для чтения (RandomAccessFile)
├─ total_chunks = ceil(file_size / chunk_size)  // макс 9999
│
├─ Для chunk_index от 0 до total_chunks-1:
│   │
│   ├─ Чтение chunk_size байт из позиции (chunk_index * chunk_size)
│   │   Последний чанк может быть меньше — это нормально
│   │
│   ├─ Base64 encode chunk bytes → data_str
│   │
│   ├─ Сборка JSON:
│   │   {
│   │     "v": 1,
│   │     "type": "image",          // или "gif"
│   │     "pub": "<ed25519_hex>",
│   │     "avatar": "<base64>",
│   │     "file_id": "<uuid_hex>",
│   │     "name": "photo.jpg",
│   │     "mime": "image/jpeg",
│   │     "size": 2457600,          // полный размер файла
│   │     "data": "<base64_chunk>",
│   │     "chunk": 0,               // индекс этого чанка
│   │     "chunks": 24              // всего чанков
│   │   }
│   │
│   ├─ Шифрование, сборка фрейма, подпись (см. пайплайн текста)
│   ├─ Отправка через _sendChain
│   │
│   └─ Отчёт о прогрессе:
│       Дельта: >3% от предыдущего → emit прогресс
│       Или: >200 мс с последнего emit → emit прогресс
│       (троттлинг чтобы не заваливать UI событиями)
│
└─ По завершении всех чанков: emit SgtpMessageSent
```

### 4.3 Сборка изображения на стороне получателя

```
Входящий MESSAGE → JSON decode → type == "image"
│
├─ Первый чанк (chunk==0):
│   ├─ Создание _PendingFile { file_id, total_chunks, received=0 }
│   ├─ Если изображение (небольшое) — храним в памяти (List<Uint8List>)
│   └─ Иначе — открываем temp-файл на диске
│
├─ Каждый следующий чанк:
│   ├─ Декодирование base64 data → bytes
│   ├─ Запись по offset = chunk_index * stride
│   │   stride = размер chunk 0 (фиксируется при первом чанке)
│   └─ received++
│
├─ Когда received == total_chunks:
│   ├─ flush() / close RandomAccessFile
│   ├─ Для in-memory: concat chunks → imageBytes
│   └─ emit SgtpMessageReceived(ChatMessage(
│         imageBytes: bytes   // или localMediaPath для крупных файлов
│       ))
│
└─ UI: Image.memory(imageBytes) или Image.file(File(localMediaPath))
```

**Важно:** чанки могут приходить не по порядку (например, чанк 3 раньше чанка 1 — relay не гарантирует порядок TCP-сессий при HTTP). Запись по абсолютному смещению (chunk_index × stride) делает сборку нечувствительной к порядку.

## 5. Голосовые сообщения (Voice)

Голосовые сообщения передаются по тому же механизму, что и изображения, но с несколькими отличиями.

### 5.1 Специфика

- **type**: `"voice"`
- **mime**: обычно `"audio/aac"` или `"audio/ogg"` в зависимости от платформы
- **Хранение у получателя**: всегда на диске (не в памяти), т.к. аудио-файлы могут быть несколько МиБ
- **Воспроизведение**: из `localMediaPath` через AudioPlayer по мере поступления (потоковое воспроизведение не реализовано — ждём полной сборки)
- **duration_ms**: опциональное поле, длительность в миллисекундах

### 5.2 JSON payload чанка голосового сообщения

```json
{
  "v": 1,
  "type": "voice",
  "pub": "ed25519_pubkey_hex",
  "avatar": "base64_png_or_null",
  "file_id": "uuid_hex",
  "name": "voice_message.aac",
  "mime": "audio/aac",
  "size": 487200,
  "data": "base64_chunk_data",
  "chunk": 0,
  "chunks": 5,
  "duration_ms": 12400
}
```

### 5.3 Пайплайн записи → отправки

```
Пользователь удерживает кнопку микрофона
│
├─ Запись в temp-файл через platform-specific audio recorder
├─ Пользователь отпускает кнопку → stop recording
├─ Temp-файл готов → sendVoice(XFile path, duration_ms)
│
└─ Тот же механизм нарезки и отправки чанков, что и для изображений
   (тип поля type = "voice", дополнительно duration_ms)
```

## 6. Видеосообщения (Video)

### 6.1 Специфика

- **type**: `"video"`
- **mime**: `"video/mp4"` или `"video/webm"`
- Видео-файлы часто большие (десятки МиБ), поэтому чанкование особенно важно
- Хранится на диске у получателя
- Thumbnail (превью первого кадра) передаётся отдельно в поле `thumbnail` (base64, опционально)

### 6.2 Оптимизация IO при отправке

При отправке больших файлов применяется **read-ahead буферизация**:

```
Вместо:
  loop: read 100 KiB → encrypt → send

Реально:
  Read-ahead buffer 30 MiB:
  ├─ Читаем 30 МиБ из файла одним вызовом IO
  ├─ Нарезаем в памяти на чанки по 100 КиБ
  ├─ Шифруем и отправляем поочерёдно
  └─ Когда буфер исчерпан → читаем следующие 30 МиБ
```

Это уменьшает количество syscall для IO в ~300 раз при почти том же потреблении памяти (30 МиБ на буфер).

### 6.3 JSON payload первого чанка видео

```json
{
  "v": 1,
  "type": "video",
  "pub": "ed25519_pubkey_hex",
  "avatar": "base64_png",
  "file_id": "uuid_hex",
  "name": "video.mp4",
  "mime": "video/mp4",
  "size": 31457280,
  "thumbnail": "base64_jpeg_preview",
  "data": "base64_first_100kib",
  "chunk": 0,
  "chunks": 300
}
```

## 7. Видеозаметки (Video Note / круглые видео)

Video Note — особый формат: короткое круглое видео, записываемое прямо в чате.

### 7.1 Специфика

- **type**: `"video_note"`
- Квадратное (обрезается в круг на UI), обычно 240×240 или 360×360 пикселей
- Длительность: как правило, до 60 секунд
- Дополнительные поля: `width`, `height`, `duration_ms`

### 7.2 JSON payload

```json

  "v": 1,
  "type": "video_note",
  "pub": "ed25519_pubkey_hex",
  "avatar": "base64_png",
  "file_id": "uuid_hex",
```

## 8. Произвольные документы и файлы

Любой файл, не попадающий в категории выше, передаётся как `"type": "file"` (или `"document"` в некоторых версиях). Механизм идентичен медиа-сообщениям, только UI отображает иконку документа вместо медиаплеера.

### 8.1 JSON payload

```json
{
  "v": 1,
  "type": "file",
  "pub": "ed25519_pubkey_hex",
  "avatar": "base64_png",
  "file_id": "uuid_hex",
  "name": "report_Q4.xlsx",
  "mime": "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet",
  "size": 5242880,
  "data": "base64_chunk",
  "chunk": 0,
  "chunks": 51
}
```

### 8.2 Сохранение у получателя

Документы сохраняются на диск в application-specific temp-директорию. Пользователь может «сохранить» файл — это копирует его в Downloads через platform-specific API.

## 9. Ответы на сообщения (Reply)

Reply-сообщения — это обычные текстовые (или медиа) сообщения с дополнительными полями:

```json
{
  "v": 1,
  "type": "text",
  "text": "Согласен!",
  "pub": "...",
  "reply_to_id": "message_uuid_hex",
  "reply_to_content": "Встречаемся в 18:00?",
  "reply_to_sender": "Алексей"
}
```

Поля `reply_to_*` — чисто информационные, не верифицируются криптографически. Подмена quoted text возможна на уровне клиента.

## 10. Прогресс отправки и статусы

Для каждого отправляемого сообщения UI получает следующие события:

| Событие | Момент |
|---|---|
| `SgtpMessageReceived` | Немедленно после нажатия «Отправить» (оптимистично) |
| `SgtpMediaProgress(progress: 0.0–1.0)` | Каждые >3% или >200 мс (только медиа) |
| `SgtpMessageReceived` | После отправки последнего чанка (финальный статус) |

**MESSAGE_FAILED flow:**

Если мастер получает MESSAGE в момент ротации ключа (между отправкой нового CHAT_KEY и получением всех CHAT_KEY_ACK), он отклоняет пакет на уровне протокола (без события в UI):

```
Мастер → MESSAGE_FAILED (зашифровано pairwise ключом, unicast)
Отправитель → получает MESSAGE_FAILED
Отправитель → MESSAGE_FAILED_ACK
Отправитель → ждёт нового CHAT_KEY → повторяет MESSAGE до 3 раз
```

## 11. Синхронизация истории (History Sync)

Когда peer переподключается или впервые вступает в активную комнату, он может запросить историю у других участников.

### 11.1 Полный пайплайн History Sync

```
[Новый/переподключившийся peer]

1. HSIR (broadcast, пустой payload)
   "Кто у вас есть история и сколько?"

2. Все peer-ы отвечают HSI (unicast):
   payload: message_count (uint64)

3. Запросчик выбирает источник с наибольшим count

4. HSR (unicast к выбранному источнику):
   payload: offset(uint64) + limit(uint64)
   "Отдай мне N сообщений начиная с offset"

5. Источник итерирует историю и отправляет HSRA батчами:

   HSRA batch #1:
     batch_number = 1
     message_count = 50
     offsets = [0, 1024, 2048, ...]  // byte offsets внутри blob
     blob = <raw MESSAGE frames concatenated>

   HSRA batch #2:
     batch_number = 2
     message_count = 50
     ...

   HSRA (EOS sentinel):
     batch_number = total_sent
     message_count = 0
     // Конец стрима

6. Запросчик:
   ├─ Парсит каждый HSRA batch
   ├─ Для каждого MESSAGE frame в blob:
   │   ├─ Расшифровывает (тем же CK, если ключ не менялся)
   │   ├─ Вставляет в локальный SQLite (дедупликация по messageUUID)
   │   └─ Emit SgtpMessageReceived(isFromHistory: true)
   └─ При EOS: emit SgtpHistorySyncComplete
```

### 11.2 Ограничения History Sync

- Не гарантировано: если все peer-ы оффлайн, история недоступна
- Relay не хранит историю вообще
- Источник отдаёт только расшифрованные (re-signed) MESSAGE-фреймы — старые ключи не сохраняются, поэтому сообщения из эпох ротации ключа до reconnect недоступны
- Медиа-чанки в истории есть, но если получатель уже собрал файл — он не будет пересобирать из истории (дедупликация по file_id)

## 12. Ротация Chat Key и обработка конфликтов

### 12.1 Плановая ротация (каждые 180 секунд)

```
[Мастер]
├─ Timer срабатывает каждые 180 секунд
├─ Генерируем новый 32-байтный random Chat Key
├─ epoch = max(prev_epoch + 1, now_unix_ms)
├─ Для каждого peer p:
│   ├─ Шифруем Chat Key pairwise ключом p
│   └─ Отправляем CHAT_KEY unicast → p
│
├─ Входящие MESSAGE игнорируются до получения всех CHAT_KEY_ACK
│   (при получении MESSAGE_FAILED_ACK отправитель повторит с новым ключом)
│
└─ После всех ACK: применяем новый ключ к исходящим сообщениям
```

### 12.2 Ротация при уходе peer-а (FIN / KICK)

```
Peer отключается → отправляет FIN (broadcast, зашифрован текущим CK)
  Нonce(8) + AEAD-tag пустого plaintext(16) = 24 байта payload

Мастер получает FIN:
  ├─ Верифицирует через дешифрование (AEAD tag подтверждает подлинность)
  ├─ Удаляет peer из списка участников
  └─ Запускает немедленную ротацию CK для оставшихся
     (чтобы ушедший peer не мог расшифровать будущие сообщения)
```

## 13. Сводная таблица типов сообщений

| Поле `type` | Описание | Хранение на получателе | Особые поля |
|---|---|---|---|
| `text` | Текст | SQLite (plaintext) | `reply_to_*` |
| `image` | Изображение (JPEG, PNG, HEIC) | Память или диск | — |
| `gif` | Анимированный GIF | Диск | — |
| `video` | Видеофайл | Диск | `thumbnail` |
| `voice` | Голосовое сообщение | Диск | `duration_ms` |
| `video_note` | Круглое видео | Диск | `width`, `height`, `duration_ms` |
| `file` | Произвольный документ | Диск | — |
| `chat_meta` | Обновление имени/аватара | Только в памяти (peer state) | `name`, `chat_avatar` |
| `message_read` | Read receipt | Не сохраняется | `msg_id` |
| `reaction` | Эмодзи-реакция | Обновляет запись в SQLite | `msg_id`, `emoji`, `add` |
