# Криптография SGTP: детальное описание

Этот документ описывает полную криптографическую модель SGTP: алгоритмы, ключевой материал, порядок операций, nonce-схемы и жизненный цикл ключей.

---

## 1. Используемые алгоритмы

| Назначение | Алгоритм | Параметры |
|---|---|---|
| Подпись каждого фрейма | **Ed25519** | 32-байтный pubkey, 64-байтная подпись |
| Эфемерный DH для pairwise | **X25519** | 32-байтный ephemeral key pair |
| Групповое шифрование (MESSAGE) | **ChaCha20-Poly1305** | 32-байтный ключ, 12-байтный nonce |
| Pairwise шифрование (CHAT_KEY, STATUS, MESSAGE_FAILED) | **ChaCha20-Poly1305** | тот же алгоритм |
| Деривация shared secret (v2) | **HKDF** или KDF(secret, room_uuid) | — |

### Почему ChaCha20-Poly1305?

- Не требует аппаратного AES-NI (работает быстро на мобильных ARM без криптоускорителя)
- AEAD: шифрование и аутентификация за одну операцию
- Nonce 96 бит (12 байт): достаточно при монотонном счётчике
- Poly1305-тег 128 бит (16 байт): добавляется к каждому ciphertext

### Почему Ed25519 + X25519?

- Ed25519 подходит для подписей: детерминированная подпись, нет nonce-leak риска
- X25519 подходит для DH: быстрый на кривой Curve25519, выходной material — 32 байта
- Оба работают на одной математической базе (Curve25519), но используются раздельно (разные ключи)

## 2. Ключевой материал: типы и жизненный цикл

### 2.1 Long-term identity key (Ed25519)

- Генерируется при первом запуске приложения, сохраняется в secure storage
- Публичный ключ (`ed25519_pub`, 32 байта) — глобальный идентификатор пользователя
- UUID участника в SGTP не равен публичному ключу; это отдельный 16-байтный идентификатор сессии
- Приватный ключ никогда не покидает устройство
- Используется: подписание каждого исходящего фрейма + whitelist-проверка входящих

### 2.2 Ephemeral X25519 key pair

- Генерируется заново для каждой сессии (при каждом подключении к комнате)
- Передаётся открыто в payload PING/PONG (32 байта pubkey)
- После завершения DH-обмена приватная часть стирается из памяти (в теории)
- Не сохраняется между сессиями → Perfect Forward Secrecy на уровне сессии

### 2.3 Pairwise shared secret

- Результат X25519 DH: `shared_secret = X25519(my_eph_priv, peer_eph_pub)`
- Размер: 32 байта (точка на кривой)
- В v1: используется напрямую как ключ ChaCha20-Poly1305
- В v2: `shared_key = KDF(shared_secret, room_uuid)` — деривация через HKDF
- Хранится в памяти (`PeerInfo.sharedKey`) на протяжении сессии
- Уникален для каждой пары (A, B) в каждой сессии

### 2.4 Chat Key (CK) — групповой симметричный ключ

- 32 байта случайных данных (`SecureRandom`)
- Генерируется мастером при каждой ротации (каждые 180 секунд)
- Распространяется каждому peer-у индивидуально через зашифрованный unicast CHAT_KEY
- Живёт ровно 180 секунд (до следующей ротации)
- После получения нового CK старый должен быть удалён из памяти

## 3. Детали DH-обмена (PING/PONG)

### 3.1 Что передаётся в PING/PONG

Payload PING и PONG идентичен (76 байт):

```
[00..31]  x25519 ephemeral public key   (32 байта)
[32..63]  ed25519 long-term public key  (32 байта)
[64..75]  "Client Hello"                (12 байт ASCII)
```

Строка `"Client Hello"` — magic для быстрой санитарной проверки без расшифровки.

Весь фрейм (Header + Payload) подписан ed25519 приватным ключом отправителя. Подпись идёт в последних 64 байтах фрейма. Получатель:

1. Верифицирует подпись, используя `ed25519_pub` из payload
2. Проверяет, что `ed25519_pub` находится в whitelist
3. Проверяет timestamp в заголовке (±30 секунд)
4. Проверяет строку `"Client Hello"` в конце payload

### 3.2 Вычисление shared secret

```
Alice (инициатор PING):
  eph_priv_A, eph_pub_A = X25519.generateKeyPair()
  Отправляет PING с eph_pub_A

Bob (получает PING, отправляет PONG):
  eph_priv_B, eph_pub_B = X25519.generateKeyPair()
  shared_secret_B = X25519(eph_priv_B, eph_pub_A)  // 32 байта
  Отправляет PONG с eph_pub_B

Alice (получает PONG):
  shared_secret_A = X25519(eph_priv_A, eph_pub_B)  // 32 байта

shared_secret_A == shared_secret_B  // математическое свойство DH
```

### 3.3 Деривация pairwise ключа (v2)

В версии 2 протокола (Flutter) сырой DH-результат не используется напрямую:

```
shared_key = HKDF(
  inputKeyMaterial = shared_secret,  // 32 байта от X25519
  salt = room_uuid,                  // 16 байт
  info = "sgtp-shared-key-v1",       // контекстная метка (UTF-8)
  length = 32
)
```

Привязка к `room_uuid` гарантирует, что один и тот же DH-обмен в разных комнатах даёт разные ключи. Контекстная метка `"sgtp-shared-key-v1"` обеспечивает domain separation и возможность смены схемы деривации в будущих версиях.

## 4. Шифрование CHAT_KEY

Мастер шифрует Chat Key для каждого peer-а индивидуально:

### 4.1 v1 формат

```
Plaintext: Chat Key (32 байта)
Key:       shared_key[peer]
Nonce:     epoch (uint64, big-endian, дополненный до 12 байт нулями слева)
Ciphertext: 32 + 16 = 48 байт (plaintext + Poly1305 tag)

Payload CHAT_KEY (v1, 56 байт):
  epoch(8) + ciphertext(48)
```

Проблема v1: если epoch повторяется (NTP-джамп назад, быстрая ротация), nonce повторяется → возможна атака на AEAD.

### 4.2 v2 формат (Flutter)

```
Plaintext: Chat Key (32 байта)
Key:       shared_key[peer]
Nonce:     random uint64, явно включается в payload
Ciphertext: 48 байт

Payload CHAT_KEY (v2, 64 байта):
  epoch(8) + nonce(8) + ciphertext(48)
```

Явный случайный nonce исключает коллизию. Получатель читает nonce прямо из payload — не надо воспроизводить epoch-based деривацию.

## 5. Шифрование MESSAGE (групповое)

### 5.1 Структура payload MESSAGE

```
[00..15]  message_uuid       (16 байт, открыто)
[16..23]  nonce              (uint64 big-endian, открыто)
[24..N ]  ciphertext         (зашифровано ChaCha20-Poly1305)
```

`message_uuid` и `nonce` — открытые метаданные. Relay и любой наблюдатель видит UUID сообщения и счётчик nonce, но не содержимое.

### 5.2 Nonce-схема для MESSAGE

```dart
// Отправитель ведёт монотонный счётчик
int _myNonce = 0;

Future<void> sendMessage(String text) async {
  final nonce = _myNonce++;  // инкремент после использования
  final ciphertext = await chaCha20Encrypt(
    plaintext: utf8.encode(jsonEncode(payload)),
    key: _chatKey,           // текущий Chat Key (32 байта)
    nonce: nonceToBytes(nonce),  // 8-байтный int → 12-байтный nonce
  );
  // ...
}
```

### 5.3 Построение 12-байтного nonce из uint64

ChaCha20-Poly1305 требует 12-байтный (96-битный) nonce. SGTP хранит nonce как uint64 (8 байт) и дополняет до 12 байт нулями:

```
nonce_96bit = [0x00, 0x00, 0x00, 0x00] + nonce_uint64_big_endian
                4 нулевых байта          8 байт big-endian
```

### 5.4 Дешифрование у получателя

```dart
// Получатель
final messageUUID = payload.sublist(0, 16);
final nonceBytes = payload.sublist(16, 24);
final ciphertext = payload.sublist(24);

final nonce96 = Uint8List(12)
  ..setRange(4, 12, nonceBytes);  // первые 4 байта = нули

final plaintext = chaCha20Decrypt(
  ciphertext: ciphertext,
  key: _chatKey,
  nonce: nonce96,
);
// AEAD-тег верифицируется внутри decrypt; при неудаче выбрасывается исключение
```

## 6. Pairwise шифрование (STATUS, MESSAGE_FAILED)

Используется для unicast-сообщений между двумя peer-ами, где содержимое не должно видеть даже другие участники комнаты.

### 6.1 v1 (Go сервер)

```
Key:   shared_key[peer]     // из DH-обмена
Nonce: timestamp из заголовка фрейма % 2^64, дополненный до 12 байт
```

Минус: nonce привязан к timestamp, а не к счётчику. Два STATUS-пакета в одну миллисекунду имеют одинаковый nonce → потенциальная коллизия.

### 6.2 v2 (Flutter)

В v2 для пакетов с pairwise шифрованием payload начинается с явного uint64 nonce:

```
[00..07]  nonce (uint64, открыто)
[08..N ]  ciphertext
```

Nonce случайный или монотонный, выбирается отправителем. Получатель читает его из начала payload до дешифровки.

## 7. Подпись фреймов (Ed25519)

### 7.1 Что подписывается

```
signed_data = header(64 байта) + payload(N байт)
signature   = Ed25519.sign(signed_data, sender_ed25519_priv)
```

Сигнатура (64 байта) идёт после payload как последний элемент фрейма. Она **не** включается в `signed_data` — это стандартная схема.

### 7.2 Верификация

```
pubkey = known_pubkey для sender_uuid
         // получен из PING/PONG.ed25519_pub, проверен через whitelist
valid = Ed25519.verify(
  message   = header + payload,
  signature = last 64 bytes of frame,
  pubkey    = pubkey
)
if (!valid) { drop frame; }
```

### 7.3 Кто верифицирует

| Участник | Верифицирует ли подписи |
|---|---|
| Relay-сервер | **Нет** — прозрачный форвардер |
| Flutter-клиент | **Да** — все входящие MESSAGE, PING, PONG |
| Go-клиент | **Да** |

Отсутствие верификации на relay означает, что атакующий с доступом к серверу может инжектировать фреймы без подписи — клиенты их отбросят, но сам факт возможности инжекции — архитектурное допущение.

## 8. Whitelist и доверие

Каждый клиент поддерживает список доверенных ed25519 публичных ключей:

```dart
Set<String> whitelist;  // Set<hex(ed25519_pub)>
```

PING и PONG принимаются только от peer-ов, чей ed25519 pubkey есть в whitelist. Это единственный механизм авторизации членства в комнате.

**Типичная конфигурация:** whitelist задаётся при создании комнаты и передаётся пользователям out-of-band (QR-код, ссылка с параметрами). При получении PING от неизвестного ключа — фрейм игнорируется (не отправляется PONG).

## 9. Известные криптографические проблемы

### 9.1 Nonce reuse в групповых сообщениях (критично)

**Проблема.** После ротации Chat Key все участники сбрасывают `_myNonce = 0`. Если два участника отправят сообщения одновременно (особенно вероятно в активной группе), оба используют `(CK, nonce=0)`.

Идентичный `(key, nonce)` → идентичный keystream ChaCha20 → XOR двух ciphertext'ов равен XOR двух plaintext'ов. Поскольку plaintext — JSON известной структуры (`{"v":1,"type":"text","text":...}`), атакующий с известным plaintext'ом (или угаданным) восстанавливает оба сообщения.

**Вектор:** Пассивный наблюдатель (relay-оператор, man-on-the-wire) записывает зашифрованные фреймы и атакует офлайн.

**Правильное решение:**
```
nonce = (sender_uuid[0:4] as uint32 big-endian) || (counter as uint32 big-endian)
// первые 4 байта из UUID отправителя + 4-байтный счётчик
// → гарантированно разные nonce у разных отправителей даже при counter=0
```

### 9.2 UUID spoofing на relay

**Проблема.** Relay регистрирует peer по `sender_uuid` из первого фрейма без проверки подписи. Атакующий может подключиться с `sender_uuid` жертвы → relay заменяет запись в `room.clients` → дальнейшие unicast-пакеты (включая CHAT_KEY) уходят атакующему.

**Правильное решение:** Relay должен верифицировать ed25519-подпись первого фрейма (INTENT) перед регистрацией peer-а.

### 9.3 Replay-атака в 30-секундном окне

**Проблема.** Клиенты проверяют `|now - timestamp| ≤ 30s`, но не отслеживают уже виденные `(message_uuid, sender_uuid)`. Атакующий может переотправить любое MESSAGE в пределах 30 секунд — relay форвардирует его снова.

**Воздействие:** Дублирование сообщений (annoyance), потенциально — повторное воспроизведение голосовых/видео. Без возможности расшифровки (у атакующего нет CK) — не критично, но неприятно.

**Решение:** Клиент должен хранить set seen `message_uuid` за последние 30–60 секунд.

### 9.4 Chat Key не зануляется в памяти

В Dart нет гарантированного способа занулить содержимое `Uint8List` до GC. После ротации старый CK остаётся в heap до сборки мусора. На рутованных устройствах или при memory dump потенциально доступен.

**Частичное решение:** `chatKey.fillRange(0, 32, 0)` сразу после замены — не гарантирует занулení на уровне VM, но снижает window exposure.

## 10. Схема ключей: сводная диаграмма

```
┌─────────────────────────────────────────────────────┐
│                    Устройство A                     │
│                                                     │
│  ed25519_priv_A ──┐                                 │
│  ed25519_pub_A  ──┼──→ Подпись каждого фрейма      │
│                   │                                 │
│  x25519_eph_priv_A ─┐                              │
│  x25519_eph_pub_A  ─┼──→ PING/PONG payload         │
│                     │                              │
│   DH(x25519_eph_priv_A, x25519_eph_pub_B)          │
│   = shared_secret_AB                               │
│   → KDF(shared_secret_AB, room_uuid)               │
│   = pairwise_key_AB  ──→ шифрование CHAT_KEY, STATUS│
│                                                     │
│  ChatKey (32 случайных байта от мастера)            │
│  = ck  ──→ шифрование всех MESSAGE                 │
└─────────────────────────────────────────────────────┘

                    Relay (не видит plaintext)
                          │
                    Форвардирует байты

┌─────────────────────────────────────────────────────┐
│                    Устройство B                     │
│  (симметричная схема)                               │
│  pairwise_key_AB = KDF(X25519(eph_priv_B, eph_pub_A), room_uuid)
│  Расшифровывает CHAT_KEY → получает ck             │
│  Расшифровывает MESSAGE с ck                        │
└─────────────────────────────────────────────────────┘
```
