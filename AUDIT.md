# Аудит: Расхождения и Уязвимости

> Дата: 2026-04-06  
> Источники: `protocol/`, `flutter/lib/`, `server/`

---

## Часть 1 — Расхождения между документацией и реализацией

### 1.1 INTENT (0x0000) отсутствует в `server/protocol/packet.go`

**Документация** (`PACKETS.md`, `WIRE_FORMAT.md`): `INTENT = 0x0000` — полноценный тип пакета.

**Реализация**:
- Flutter: `PacketType.intent = 0x0000` ✓
- `server/protocol/packet.go`: константа `TypeIntent` **не определена**. Первый входящий фрейм на сервере принимается как "intent" без проверки `packet_type`. Сервер не валидирует, что первый кадр действительно INTENT.

---

### 1.2 Версия протокола: сервер всегда шлёт v1, Flutter — v2

**Документация** (`COMPATIBILITY.md`): расхождение упомянуто.

**Реализация**:
- `server/protocol/packet.go`: `ProtocolVersion = 0x0001`
- `flutter/lib/core/constants.dart`: `version = 0x0002`

Сервер является чистым relay и никогда не создаёт фреймы, поэтому конфликта нет — но любой будущий Go-клиент, использующий `server/protocol`, будет отправлять v1, что несовместимо с v2-расширениями (CHAT_KEY explicit nonce).

---

### 1.3 CHAT_REQUEST: расширенный Flutter-формат не отражён в Go-парсере

**Документация** (`PACKETS.md`): «Flutter extension: дополнительно `chat_name` и `avatar`».

**Реализация**:
- Flutter `buildChatRequest`: payload = `count(8) + UUIDs + nameLen(4) + name(UTF-8) + avatarLen(4) + avatar`.
- `server/protocol/unmarshalChatRequest`: парсит только `count + UUIDs`, расширенный хвост игнорируется (без ошибки).

Сервер-relay просто пересылает байты, поэтому на практике это работает. Но Go-парсер создаёт неполный объект `ChatRequest` и не сообщает об ошибке — потенциальная проблема для будущего Go-клиента.

---

### 1.4 CHAT_KEY v2 не реализован в `server/protocol/packet.go`

**Документация** (`MESSAGING_AND_KEYS.md`, `COMPATIBILITY.md`): v2 добавляет явный `nonce(8)` между `epoch` и `ciphertext`.

**Реализация**:
- Flutter `buildChatKey`: при `version >= 0x0002` payload = `epoch(8) + nonce(8) + ciphertext(48)` = 64 байта.
- `server/protocol/ChatKey.Marshal()`: всегда генерирует `epoch(8) + ciphertext` = v1.
- `unmarshalChatKey`: не читает nonce из v2-пакета.

Go-парсер при получении v2 CHAT_KEY неверно интерпретирует первые 8 байт ciphertext как начало зашифрованного ключа.

---

### 1.5 HSRA EOS: недокументированная семантика поля `batch_number`

**Документация** (`PACKETS.md`): «`HSRA.message_count = 0` означает end-of-stream».

**Реализация**:
- Flutter `buildHsraEos`: первый uint64 (поле `batch_number`) = `totalSent`, второй uint64 (`message_count`) = 0.
- `server/protocol/HSRA.Marshal()` при EOS не оговорено, что `BatchNumber` несёт `totalSent`.

Контракт «при EOS, batch_number = общее число отправленных пакетов» нигде явно не задокументирован. Это скрытый протокол.

---

### 1.6 Discovery: документация `reference/server_spec.md` не описывает HTTP JSON endpoint

**Реализация**: Flutter делает `GET /sgtp/discovery` (HTTP JSON), не TCP 25-байтный баннер для discovery.

**Документация** `reference/server_spec.md`: описывает только legacy 25-байтный TCP binary формат.  
`TRANSPORTS_AND_DISCOVERY.md`: корректно описывает оба варианта. Но `reference/server_spec.md` устарел.

---

### 1.7 Сервер не проверяет тип первого пакета

**Документация** (`HANDSHAKE.md`): «Новый клиент отправляет `INTENT`».

**Реализация** (`server/server.go`): принимает **любой** первый фрейм как "intent", не проверяя `packet_type`. Любой пакет (PING, MESSAGE, etc.) откроет соединение в комнате.

---

### 1.8 Сервер не валидирует подпись и timestamp

**Документация** (`WIRE_FORMAT.md`, раздел «Валидации»): перечислены проверки — timestamp window, длина, подпись.

**Реализация** (`server/server.go`): явно задокументировано: «The server does NOT decrypt, validate signatures, or maintain session state». Сервер проверяет только `payload_length <= MaxPayloadLength`. Остальные валидации — на клиенте.

Расхождение: протокол описывает валидации как требования к «получателю», не уточняя «relay-сервер vs клиент». Это требует уточнения в документации.

---

### 1.9 Размер аватара в CHAT_REQUEST не ограничен на уровне парсера

**Документация** (comment в `frame_builder.dart`): «`avatar_bytes (max 4KB)`».

**Реализация**: ни `buildChatRequest`, ни `unmarshalChatRequest` на Go-стороне не проверяют лимит аватара. Только имя ограничено (`≤ 255 UTF-8 байт`).

## Часть 2 — Уязвимости в протоколе

### 2.1 🔴 Критическая: Nonce reuse (ChaCha20-Poly1305) при группе ≥ 2 клиентов

**Где**: `sgtp_client.dart`, `_issueCK()` / `_onChatKey()`.

Все участники комнаты разделяют один `Chat Key` (CK). При каждой ротации каждый клиент сбрасывает `_myNonce = 0` и начинает с nonce=0. Если два клиента отправляют сообщение после одной и той же ротации CK, они используют **одинаковый (key, nonce=0)**:

```
Client A: encrypt(plaintextA, CK, nonce=0)
Client B: encrypt(plaintextB, CK, nonce=0)
```

В ChaCha20-Poly1305 одинаковый `(key, nonce)` порождает **одинаковый keystream**. Зная оба ciphertext, атакующий получает `plaintextA XOR plaintextB`. Это позволяет восстановить оба открытых текста при наличии шаблонов (JSON-формат).

**Исправление**: включать UUID отправителя в nonce (например, первые 4 байта UUID + 4 байта счётчика), либо использовать детерминированный namespace для nonce по sender UUID.

---

### 2.2 🔴 Критическая: UUID spoofing на уровне relay-сервера

**Где**: `server/server.go`, функция `r.add(cn)` / routing.

Сервер регистрирует клиента в `room.clients[senderID]`, где `senderID` берётся напрямую из поля `sender_uuid` intent-фрейма — **без проверки подписи**. Злоумышленник может:

1. Подключиться с `sender_uuid = UUID жертвы`.
2. Сервер добавит атакующего в `room.clients[UUID_жертвы]`.
3. Unicast-пакеты (`CHAT_KEY`, `STATUS`, `MESSAGE_FAILED`) адресованные UUID жертвы пойдут атакующему.
4. Реальная жертва получит `replaced.Close()` и будет отключена (строка `r.add` возвращает replaced conn).

Атака позволяет перехватить CHAT_KEY и получить доступ к истории, а также вытолкнуть легитимного пользователя.

**Частичная защита**: клиент перед принятием CHAT_KEY проверяет подпись мастера. Но это не защищает от перехвата пакетов в промежутке.

**Исправление**: сервер должен хранить подпись INTENT и сверять последующие пакеты от того же UUID с тем же ed25519 ключом, либо применять challenge-response на уровне соединения.

---

### 2.3 🔴 Критическая: TLS-сертификат игнорируется при HTTP discovery

**Где**: `flutter/lib/features/messaging/data/transport/server_discovery_http_client_io.dart`.

```dart
..badCertificateCallback = (_, __, ___) => true;
```

Discovery-запрос `GET /sgtp/discovery` выполняется по HTTPS с **отключённой проверкой сертификата**. MITM-атакующий в локальной сети может:

1. Перехватить HTTPS-запрос discovery.
2. Вернуть поддельный JSON с адресом вредоносного relay.
3. Клиент подключится к злоумышленнику.

Последствия: полный перехват трафика комнаты (до того как клиент проверит подписи PING/PONG).

**Исправление**: убрать `badCertificateCallback`, использовать системный trust store или certificate pinning.

---

### 2.4 🟠 Высокая: Replay attack — отсутствие защиты от повторной доставки фреймов

**Где**: клиент, `_dispatch()`.

Проверяется только `timestamp ∈ [now-30s, now+30s]`. Уникальность фрейма (по `messageUUID` или подписи) **не отслеживается**. Атакующий, способный перехватить трафик, может повторно отправить любой MESSAGE в течение 30-секундного окна.

Для текущей архитектуры (relay-сервер без памяти) атакующий должен быть в позиции MITM или иметь доступ к серверу. Но в случае компрометации relay это становится практичным.

**Исправление**: клиент должен хранить набор уже обработанных `messageUUID` (с TTL 30s) и отбрасывать дубликаты.

---

### 2.5 🟠 Высокая: Отсутствие rate limiting и room limit на сервере

**Где**: `server/server.go`.

Нет ограничений:
- Числа подключений с одного IP.
- Числа комнат.
- Числа участников в комнате.
- Частоты фреймов.

Атакующий может открыть тысячи соединений, исчерпав память сервера (`rooms` map растёт неограниченно), или создать широковещательный шторм в большой комнате.

Буфер `sendCh` (64 пакета, `sendTimeout = 250ms`) ограничивает нагрузку на одного клиента, но не на сервер в целом.

---

### 2.6 🟠 Высокая: INFO-ответ не аутентифицирован относительно комнаты

**Где**: `HANDSHAKE.md`, `sgtp_client.dart`.

Мастер в `INFO`-ответе возвращает список UUID. Клиент доверяет этому списку и pingует всех указанных участников. Скомпрометированный или вредоносный мастер может:

1. Включить в список произвольные UUID.
2. Новый клиент начнёт handshake с ними, раскрывая свой x25519 публичный ключ.
3. При наличии whitelist PING будет отброшен. Но если whitelist не настроен — возможен нежелательный DH exchange.

---

### 2.7 🟡 Средняя: CHAT_KEY epoch как nonce в v1 — слабость при быстрых ротациях

**Где**: `_issueCK()`, v1 путь.

```dart
final nonce = version >= 0x0002 ? _randomUint64() : _chatEpoch;
```

`_chatEpoch = timestamp_мс`. Если мастер ротирует CK дважды в одну миллисекунду (например, из-за добавления нового участника сразу после ротации), `_chatEpoch` инкрементируется на 1. Но если системные часы не монотонны (NTP прыжок назад), epoch может не измениться или уменьшиться.

При совпадении `(sharedKey, epoch_nonce)` для двух разных CK — nonce reuse в pairwise-шифровании.

v2 использует `_randomUint64()` — проблема устранена для v2.

---

### 2.8 🟡 Средняя: KICK_REQUEST не авторизован — возможна нагрузочная атака

**Где**: `HISTORY_FIN_KICK.md`, клиент.

Любой аутентифицированный участник комнаты может отправить `KICK_REQUEST` с произвольным `target_uuid`. Мастер получает запрос и обязан пинговать target. При массовой отправке KICK_REQUEST'ов злоумышленник может создать непрерывный поток PING-операций на мастере.

Протокол не ограничивает частоту KICK_REQUEST от одного участника.

---

### 2.9 🟡 Средняя: Открытый `sender_uuid` позволяет отслеживать участников

**Где**: wire format.

`sender_uuid` и `receiver_uuid` в заголовке передаются в открытом виде (не шифруются). Пассивный наблюдатель (включая оператора relay-сервера) видит граф общения: кто кому пишет, когда. Содержимое зашифровано, но metadata (участники, временны́е паттерны, объёмы) раскрыты.

---

### 2.10 🟡 Средняя: Чат-имя и аватар в MESSAGE payload (ненадёжная атрибуция)

**Где**: `sendMessage()` в `sgtp_client.dart`.

JSON-payload включает поле `pub` (ed25519 pubkey) и иногда поле `avatar`. Эти значения берутся из локального состояния клиента и **не верифицируются отдельно** получателем относительно PING-рукопожатия. Если атакующий знает чужой ed25519 публичный ключ, он может включить его в свой `pub`-field в MESSAGE, подписав MESSAGE своим ключом. Получатель проверяет подпись фрейма по ключу из `_peers`, что корректно. Но связка «автор MESSAGE = владелец `pub`-поля внутри JSON» не доказывается криптографически.

---

## Часть 3 — Уязвимости в реализации (без протокольного покрытия)

### 3.1 🟠 Высокая: Небезопасная генерация Chat Key на клиенте

**Где**: `_issueCK()`.

```dart
final key = Uint8List.fromList(
    List.generate(32, (_) => _secureRandom.nextInt(256)));
```

Используется `Random.secure()` — это криптографически стойкий PRNG в Dart. ✓  
Но операция `List.generate` создаёт промежуточные объекты в heap. При GC-давлении ключ может оставаться в памяти в старых поколениях объектов. Нет явного обнуления после использования.

---

### 3.2 🟡 Средняя: Chunk-upload — `_firstChunkSize` используется как stride без проверки

**Где**: `_PendingFile.writeChunk()`.

```dart
final offset = _firstChunkSize > 0 ? ci * _firstChunkSize : 0;
await raf.setPosition(offset);
```

Если chunk 0 отсутствует (`_firstChunkSize == 0`), все чанки записываются в offset=0 — данные затираются. При out-of-order доставке (что невозможно в TCP, но возможно после рефакторинга) будет запись в неверные позиции.

---

### 3.3 🟡 Средняя: Chat name длина — сервер не проверяет, клиент — только при отправке

**Где**: `buildChatRequest` в `frame_builder.dart`.

```dart
if (nameUtf8.length > 255) throw ArgumentError('Chat name must be ≤ 255 UTF-8 bytes');
```

Ограничение 255 байт проверяется **только при отправке**. При получении `frame_parser.dart` (метод `chatRequestName`) не проверяет длину имени, а вызов `utf8.decode` может получить произвольно длинную строку. Возможна передача чат-имени размером несколько мегабайт.

---

### 3.4 🟢 Низкая: `buildIntentFrame` не подписывается

**Где**: `frame_builder.dart`, `buildIntentFrame`.

Intent frame создаётся с нулевой подписью (нет вызова sign). Signature-поле будет заполнено нулями. Это ожидаемое поведение (INTENT — триггер, не требующий аутентификации) и не противоречит протоколу явно, но документация не оговаривает, что INTENT может иметь нулевую подпись. Принимающий клиент проверяет подпись PING/PONG, но не INTENT.
